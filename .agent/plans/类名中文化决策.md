# 类名中文化决策

## 目标

在“字段已全中文、类名仍为英文”的现状下，评估是否继续把 `src/models.py` 中的核心类名改成中文，并给出可执行决策。

当前讨论对象以这些核心类为主：

- `DocumentData`
- `DocumentProfile`
- `PageRecord`
- `SectionRecord`
- `TableRecord`
- `NumericParameter`
- `RuleRecord`
- `StandardReference`

## 当前事实

- 字段层已经中文化，下游主链也基本以中文字段访问。
- 类名目前主要存在于：
  - `src/models.py`
  - `src/parser.py / normalizer.py / reviewer.py / fixer.py / exporter.py`
  - `tests/`
  - 少量类型注解、构造代码、辅助访问层
- 当前痛点不是“类名影响功能”，而是“概念口径不完全统一”。
- 当前更高优先级的问题仍然是：
  - reviewer issue 命中条件复核
  - OCR 文本质量与表格单元格对齐精度
  - LLM 中文化主路径占比继续提升

## 方案一：全量中文化

做法：

- 把 `src/models.py` 的 dataclass 类名整体改成中文。
- 全仓同步修改 import、类型注解、构造调用和测试。
- 不保留英文兼容别名，直接一次性切换。

收益：

- 代码符号与字段层口径完全一致。
- 新同学阅读“领域模型”时更贴近业务语义。
- `First Principles` 的“全中文契约”从字段层推进到类型层。

风险：

- 改动面最大，属于高扩散重命名。
- IDE 搜索、类型提示、静态检查、第三方生态习惯都会受影响。
- 未来若接入更多英文生态库、脚手架、自动生成工具，摩擦会更大。
- 回归成本高，且收益主要是“风格一致性”，不是功能提升。

适用条件：

- 团队明确把“代码标识中文化”当成长期规范。
- 可以接受一次较大的重命名回归窗口。

综合判断：

- **不推荐作为当前阶段选项。**

## 方案二：只改核心类

做法：

- 只中文化最核心、最高频暴露的 3 到 5 个类，例如：
  - `DocumentData`
  - `DocumentProfile`
  - `PageRecord`
  - `SectionRecord`
  - `TableRecord`
- 其余值对象和记录类暂时保留英文。

收益：

- 比全量改动小，能改善“最常见类型名仍是英文”的割裂感。
- 用户沟通、设计文档、代码 walkthrough 时更统一。

风险：

- 会出现“类名一半中文、一半英文”的新混搭状态。
- 认知复杂度不一定下降，可能只是把边界换了位置。
- 仍需修改大量 import / 类型注解 / 测试。

适用条件：

- 团队非常在意 `DocumentData` 等核心概念的中文一致性。
- 但暂时不愿承担全量重命名成本。

综合判断：

- **可作为折中方案，但不是最优。**

## 方案三：保持现状

做法：

- 保留英文类名。
- 继续坚持：
  - 字段全中文
  - 输出全中文
  - 访问契约全中文
- 把“类名中文化”视为风格决策，不纳入当前主线。

收益：

- 风险最低，不打断当前功能主线。
- 与 Python 生态、IDE、类型系统、库交互更自然。
- 团队主要精力可以继续投向质量问题，而不是符号重命名。

风险：

- 代码符号层仍然存在“英文类名 + 中文字段”的双口径。
- 对强中文化风格偏好的成员来说，不够彻底。

适用条件：

- 当前目标优先级是“准确率、稳定性、可交付性”。
- 团队接受“类型名英文、字段与输出中文”的长期状态。

综合判断：

- **推荐作为当前决策。**

## 推荐结论

当前建议选择：**方案三，保持现状。**

原因只有两条，但足够决定：

1. 这项改动的主要收益是“风格一致性”，不是“功能正确性”。
2. 当前项目更值钱的工作还在 reviewer 命中条件、OCR 文本质量、表格单元格对齐这些真正影响评分和交付质量的地方。

## 触发重开条件

如果后面出现下面任一情况，可以重新评估是否进入方案二：

- 类名频繁出现在用户可见配置、插件接口、自动生成文档里，造成明显理解成本。
- 团队正式确定“代码标识中文化”是长期规范，而不是局部偏好。
- 主链质量工作基本收口，项目进入以一致性和维护体验为主的整理阶段。

## 若用户坚持要改，建议落地顺序

1. 先做方案二，不直接做方案一。
2. 第一批只处理 `DocumentData / DocumentProfile / PageRecord / SectionRecord / TableRecord`。
3. 每批改动都必须附带全量类型注解搜索、回归测试和输出契约验收。
4. 如果方案二实际收益不明显，就停止，不继续扩到全量。
