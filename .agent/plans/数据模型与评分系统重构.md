# 数据模型与评分系统重构

> 角色：Plan Agent  
> 范围：`src/models.py`、`src/structured_access.py`、`src/summarizer.py`、`src/tagger.py`、`src/reviewer.py`、`src/pipeline.py`、`src/fixer.py`、`src/exporter.py`、`src/app.py`、`web/runner.py`  
> 依据：`First Principles.md` 第 6 节、第 8 节、第 10 节、第 11 节  
> 说明：本设计以当前仓库真实状态为准，不沿用过期上下文中的旧假设。

## 1. 模块目标

### 1A. 数据模型中文契约收口

当前仓库里，`DocumentData` 与 `DocumentProfile` 主字段已经基本中文化，`v2` 字段也已不在 `src/models.py` 中。真正未收口的点，不再是 dataclass 主属性本身，而是“访问层与中间结构仍混用英文键名”：

- `src/structured_access.py` 返回的 dict 仍使用英文键：
  `number / title / heading / level / parent_number / body / part / id / name / value_text / unit / condition / section_name / source_table / source_item / source_refs / rule_type / content / code / family / model / series / display_name`
- 下游消费这些英文键的模块有：
  `src/summarizer.py`、`src/tagger.py`、`src/reviewer.py`、`src/llm_refiner.py`

本轮 Commit A 的目标不是再次重命名 `DocumentData`，而是把“访问层输出契约”也统一到中文，形成：

`PDF -> parser -> DocumentData（中文字段） -> structured_access（中文键） -> summarizer/tagger/reviewer -> exporter`

这样才真正符合 `First Principles.md` 第 6 节“只维护一套数据模型、字段名使用中文”的方向，也回应第 11 节最后一条“内部数据模型与访问层继续收口到单一 DocumentData 中文契约”。

### 1B. 评分系统收口

`src/reviewer.py` 当前已经完成过一轮重构：

- 重复函数定义已经清掉，当前 `def` 名称全局唯一
- 已引入 35/40/25 三维评分
- 已引入 `REDLINE_CAP = 74.0`

但仍有三类偏差没有完全对齐 `First Principles.md` 第 10 节：

1. 红线规则不止 3 条，且存在 `59.99 / 74.99 / 79.99` 等多种 cap
2. `review.json` 新旧字段并存，`pipeline.py / app.py / web/runner.py` 仍消费旧键：
   `最终总评 / 最终通过 / 红线是否触发`
3. `reviewer.py` 的问题列表、红线列表、分项评分结构还没有完全收口到单一中文契约

本轮 Commit B 的目标是把评分系统收口到一个严格、可解释、可贯通的契约：

- 评分维度固定为 `35 / 40 / 25`
- 红线固定为 FP 第 10 节的 3 条
- 通过条件固定为 `总分 >= 85 且无红线`
- `review.json`、`review_rounds.json`、`pipeline.py`、`fixer.py`、`app.py`、`web/runner.py` 全部只认一套中文键

## 3. 数据流图

### 3.1 数据模型链路

```text
PDF
  -> parser.parse()
  -> DocumentData（中文字段）
  -> structured_access（中文键）
  -> summarizer / tagger / reviewer
  -> exporter
  -> output/*.json
```

### 3.2 评分链路

```text
document + markdown + summary + tags
  -> reviewer 子检查
  -> 问题清单
  -> 三维扣分（35 / 40 / 25）
  -> 红线检查（仅 3 条）
  -> review.json
  -> pipeline / fixer / app / web.runner 消费
```

## 4. 分步实现计划

### Commit A：访问层中文化

#### A1. 确认 models 层不再重复改名

要做什么：

- 只验证 `DocumentData`、`DocumentProfile` 当前字段是否已中文化
- 不对 `models.py` 做无意义重命名

- `src/models.py` 中不再存在 `metadata / raw_pages / sections / tables / numeric_parameters / rules / inspections / standards / blocks / profile`
- 不再存在 `pages_v2 / nodes_v2 / products_v2 / parameter_facts_v2 / rule_facts_v2 / standard_facts_v2 / ParsedDocument`

#### A2. 改 `structured_access.py`

要做什么：

- 把 `get_section_entries()`、`get_parameter_entries()`、`get_rule_entries()`、`get_standard_entries()`、`get_product_entries()` 返回 dict 的键全部改成中文

如何验证：

- 对 `structured_access.py` 搜索不到上述英文键字面量

#### A3. 同步下游消费点

要做什么：

- `src/summarizer.py`
- `src/tagger.py`
- `src/reviewer.py`
- `src/llm_refiner.py`

全部改用中文键访问

如何验证：

- 搜索不到：
  `["name"]`、`["value_text"]`、`["unit"]`、`["condition"]`、`["section_name"]`、`["source_table"]`、`["source_item"]`、`["rule_type"]`、`["content"]`、`["code"]`、`["family"]`、`["model"]`、`["series"]`、`["display_name"]`

#### A4. 静态校验

要做什么：

如何验证：

- 无语法错误

### Commit B：评分系统收口

#### B1. 收紧 `reviewer.py` 红线定义

要做什么：

- 只保留 FP 第 10 节的 3 条红线：
  - 正文主链缺失
  - 表格存在但数值型参数为空
  - 文本层不足需要 OCR 且未恢复正文主链
- 统一红线上限为 `74`

如何验证：

- `reviewer.py` 中不再出现 `59.99 / 74.99 / 79.99`
- 红线列表只会产出 3 种名称

#### B2. 收口 `review_outputs()` 返回字段

要做什么：

- 删除旧兼容键：
  `最终总评`、`最终通过`、`红线是否触发`
- 保留单一中文键：
  `总分`、`是否通过`、`红线触发`

如何验证：

- `review.json` 中不再出现旧字段

#### B3. 同步消费方

要做什么：

- `src/pipeline.py`
- `src/fixer.py`
- `app.py`
- `web/runner.py`

改为只读新键

如何验证：

- 以上文件搜索不到：
  `最终总评`、`最终通过`、`红线是否触发`

#### B4. 保持修正路由稳定

要做什么：

- `fixer.py` 继续从 `问题清单` 和 `红线列表` 读动作，不引入第二套字段
- 如果问题类型常量保留原样，则只改字段名，不改动作映射语义

如何验证：

- `classify_fix_actions()` 和 `apply_fixes()` 不因字段改名失效

## 7. 风险与开放问题

### 7.1 已确认风险

- 旧计划文档与当前代码状态不一致，不能按旧映射机械替换
- `reviewer.py` 已经做过一轮收口，如果再次大改必须先保留其“问题常量 -> 修复动作”的兼容关系
- `structured_access.py` 的英文键虽然不对外落盘，但被多个模块依赖，改动会有联动面

### 7.2 本轮不做

- 不修改 `First Principles.md`
  原因：按规则必须先把实现收口，再把 diff 给用户确认
- 不改 Web UI 展示文案
  原因：当前需求主线是模型契约和评分系统
- 不引入兼容 shim
  原因：`fp-code-agent` 明确禁止保留向后兼容的旧字段

### 7.3 开放问题

- 仓库里当前只存在 `fp-plan-agent` 和 `fp-code-agent` 两个 skill 文件，`fp-review-code-agent`、`fp-review-output-agent` 路径不存在  
  本轮默认按其预期职责手工执行“代码审查”和“输出审查”，不额外创建新 skill 文件
