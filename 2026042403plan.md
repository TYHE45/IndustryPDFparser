**目标：** 让 `process_log.json` 新增两个哈希字段，下一轮慢速基线飘移可一眼归因到"prompt 变了 / reviewer 规则变了 / PDF 变了"。

### 3.1：新增 `src/config_signatures.py`

新建小模块，集中放哈希生成函数。单独一个模块更好测、不污染 `pipeline.py`。

```python
# src/config_signatures.py
from __future__ import annotations
import hashlib
from src.summarizer import _summary_system_prompt
from src.tagger import _tag_system_prompt
from src.reviewer import ISSUE_DEDUCTIONS

def _short_hash(s: str) -> str:
    return hashlib.sha256(s.encode("utf-8")).hexdigest()[:8]

def prompt_signature() -> str:
    """summarizer + tagger system prompt 的联合 8-hex 签名。"""
    joined = _summary_system_prompt() + "\n---\n" + _tag_system_prompt()
    return _short_hash(joined)

def reviewer_signature() -> str:
    """ISSUE_DEDUCTIONS 按 key 排序后的规范化 8-hex 签名。"""
    canon = "\n".join(
        f"{k}|{v[0]}|{v[1]:.2f}" for k, v in sorted(ISSUE_DEDUCTIONS.items())
    )
    return _short_hash(canon)
```

- **为什么 8-hex 而非完整 64-hex**：人眼可扫 / git log 可目视对比；8-hex 碰撞概率 2^32，对"同一项目的几轮 plan 内签名集合"足够
- **为什么不 hash 整个 reviewer.py 源码**：`_review_*` 函数体改动会换签名但实际 review 行为未必变；锁住 `ISSUE_DEDUCTIONS` 这个真值字典即可；若未来想覆盖更细（如 reviewer 里的正则常量），再分离 `reviewer_rules_signature()` / `reviewer_code_signature()` 两个哈希
