## 二、本轮新发现的缺陷（2026-04-24 盘点）

### 🔴 最严重 — 工程卫生红灯

**17 个文件 / 2 个新 src 模块 / 5 个新测试 / 2 个新 plan 文档 / FP 自己的更新 / 5 个 validation 目录**全部堆在 working tree 未提交。上次 commit 还是 4-23 的 `c6306c5`。`git diff --stat` 显示 **+772 / -81**。

- `src/ocr.py` 一个文件就 +503 行
- 跨 7 个阶段的改动混在一次 diff 里
- 任何误操作会一次性丢失整轮成果
- 现在 review / rollback 已经无法按阶段切片

### 🟡 严重 — 缺少分数快照回归保护

19 项测试覆盖契约 / 合同 / 运行计划，但没有一项锁"具体样例真实得分"。如果某个改动让 `SN544-1` 从 100 降到 92 分，测试全会过。靠人肉重跑样例才能发现。

### 🟡 严重 — 样本语料仍然窄

`input/industry_standard/` 绝大多数是 SN5xx 中文系列 + 1 份英文 SN775 + 1 份 CB 589。**缺**：产品目录 / 图纸密 PDF / 纯表格扫描 / 原版 DIN/ISO / 德文标准 / 水印污染样本。

§0 工作纪律第 3 条（通用性）**目前没有数据支撑**——正则 / 阈值 / blacklist 只在 SN5xx 下被验证过。

### 🟠 中等 — OCR 子系统单测深度不足

`src/ocr.py` +503 行新能力（懒加载 + 运行计划 + 表格结构 + 软超时），对应测试只有 6 条。下列路径未被断言：
- 引擎不可用降级
- 批次中途软超时恢复
- 表格结构与 OCR 文本合并边界
- 多页批次对齐失败

### 🟠 中等 — safety-net 触发率缺乏量化

`text_localization` 已从主动翻译退位为带 warning 的安全网，但仍然是"看日志"。没有任何地方把"本次运行触发了 N 次 safety-net"落进 `process_log.json` 或 `review.json`。

**后果：** B2 的关键 KPI（主路径占比）没有可量化证据；待办里"压低 warning 触发率"只能靠 grep 日志，不能进回归测试。

### （FP 既有）— reviewer 命中条件未真正改进

P0 反测 + B1 证明：7 份样例里 5 份 0 问题 → 100 分。瓶颈不是"扣分数字太轻"，是"子检查没命中"。B1 已经用实验证明继续调数字没用。
